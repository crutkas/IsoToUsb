Concept A v2 — two floating cards

This iteration collapses the previous three-pane Workshop into two surfaces: a Setup card on the left and one unified Work card on the right. The Work card always owns both the live log and pipeline, so the build's narration and its phase position are locked together visually.

With the redundant success message inside Setup removed, Done is communicated by the bottom action bar status pill, progress, and current-operation caption while the Setup card's Done slot stays intentionally quiet.

V1 · Concept A v2

Phase rail + log

A slim 72 px phase rail sits on the left edge of the Work card and the log fills the rest. The rail is the table of contents; the log is the page.

Done keeps the Work card mildly elevated because the user's last attention was there; the Setup bottom slot becomes a quiet readiness badge, not a banner.

Transition cues

  • Setup: left Setup card lifts; Work card flattens and desaturates, but the rail still previews the six phases.
  • Building: Work card takes the 16 px focus shadow; the active rail node glows and the log streams beside it.
  • Done: success green replaces accent blue; Work settles to a mild shadow while the bottom bar owns the completed state.

Pros / cons

  • Low WinUI risk: Grid with a 72 px column, ItemsControl rail, existing LogLines and Phases.
  • Excellent casual scannability: users see the current phase without reading every log line.
  • Accessible if rail nodes expose phase Name/Detail; abbreviations need AutomationProperties.Name.

V2 · Concept A v2

Pipeline header strip + log

The Work card starts with a GitHub Actions-style breadcrumb strip of all six phases; the log fills the large area below. It reads like one live run surface with a compact progress header.

Done returns both cards to a neutral resting elevation; the Setup bottom slot offers a restrained Build another affordance while the bottom bar remains the source of truth.

Transition cues

  • Setup: Setup card is raised; the Work strip is present but quiet so the future path is visible.
  • Building: Work card lifts; the active chip gets the blue halo and completed chips turn green.
  • Done: both cards relax to rest elevation, reducing visual urgency once the run is settled.

Pros / cons

  • WinUI implementation is straightforward: horizontal ItemsControl/ItemsRepeater above the existing log view.
  • Very screen-reader friendly if chips are ordered and each has status text.
  • Casual users may scan the strip faster than a rail, but long phase names need careful trimming at 1240 px.

V4 · Concept A v2

Vertical split inside the card

The Work card is internally divided: pipeline list in the top ~40%, log in the bottom ~60%. Both are always visible, but the single outer card surface makes them one concept.

Done leaves Work mildly elevated so the finished run remains the artifact; Setup quietly resets to a ready-for-next helper rather than repeating completion copy.

Transition cues

  • Setup: Setup card lifts; top pipeline and lower log flatten together inside one Work surface.
  • Building: Work card lifts as a whole; the divider stays still while phase rows and log content update.
  • Done: Work keeps a gentle success-tinted rest shadow, emphasizing that the artifact is the completed run record.

Pros / cons

  • Lowest cognitive risk: both pipeline details and log lines are visible at the same time.
  • Moderate WinUI risk: fixed top height plus ScrollViewer bottom must handle small windows and text scaling.
  • Good accessibility because it preserves the current pipeline list shape; less efficient for log-heavy users than V1/V2.

Designer recommendation

Choose V1 · Phase rail + log. It is the strongest two-card composition because the log remains the hero while the phase rail gives continuous location context without stealing width or hiding pipeline status behind a tab.

It is also the cleanest WinUI 3 path: existing Phases bind into a narrow ItemsControl, existing LogLines keep filling the log, and the elevation shift can be done with ThemeShadow, opacity, and a ≤180 ms RenderTransform.
