Elevation indicator — make admin runtime unmistakable

The UI process runs as asInvoker; a separate worker process is spawned via UAC only when the user clicks Start build. Today, that worker's existence is signalled by an 11 pt secondary-text label "🛡 Elevated worker" tucked in the bottom-left footer — visually identical whether the worker is alive or not. The user wants much louder presence while the worker is actually running (and quiet calm when it's not).

What stays the same

The bottom-left position concept (the user likes it), the phase rail + log as the visual focus, and the asInvoker manifest. No layout reflow on state change.

What must change

When IsBusy=true (worker spawned and pipeline running), the elevation status must be impossible to miss — full-width tint, animated cue, or a colored window-chrome accent. When idle, it should fade to a quiet hint.

Accent / elevation blue (#0067C0 light, #4CC2FF dark) Warning / target drive (#C04C00 / #FFB454)  Shield (UE7EF, Segoe Fluent Icons)  Permissions (UE730)
Option A

Prominent footer band

The existing bottom-left footer grows in place into a full-width tinted band the moment the worker spawns — same anchor the user already likes, but loud.

Behavior

  • Idle → 22 px slim caption: "asInvoker · will elevate on Start".
  • Active → 38 px tinted band slides up in place, no other layout shift.
  • Subtle 3.4 s sheen sweeps L→R while running; pulse dot at right.
  • Band carries the target drive callout (the most destructive bit of context).

Dark-mode parity & a11y

  • Light: accent #0067C0 over warm grey; dark: #4CC2FF over #202020 — both ≥ 4.5:1 on body text.
  • Sheen + pulse respect prefers-reduced-motion (static fill only).
  • The band itself is a live region so screen readers announce "Elevated worker active — writing to Disk 1, USB DISK 3.0".

Option B

Floating elevation chip

A solid-fill accent chip floats on the top-right of the Work card while the worker is alive — like the macOS recording indicator, always in peripheral vision wherever the user looks.

Behavior

  • Idle → footer says "asInvoker"; no chip anywhere.
  • Active → chip fades in over the Log header. Pulse dot ticks at 1.6 s.
  • Hover reveals tooltip "PID 8421 · started 14:02:11 · phase Wipe".
  • Footer also brightens to a soft accent fill so peripheral cues exist top + bottom.

Dark-mode parity & a11y

  • Solid accent fill in both themes (white text) holds 8:1+ contrast.
  • Chip never overlaps the cancel/start row — sits inside Log header padding.
  • Lightest footprint of the three; survives if the window is narrow.

Option C

Title-bar tint + reinforced footer

The entire window chrome announces the state — title bar tints accent-blue with a shield badge, footer carries a matching pill. Visible even when the window is minimized to taskbar preview.

Behavior

  • Uses AppWindowTitleBar.BackgroundColor + ButtonForegroundColor APIs on the actual title bar (no client-area emulation).
  • Title bar shows shield icon, "ELEVATED" pill with pulse dot, app title.
  • Footer keeps the user's preferred bottom-left anchor with a shield pill + target drive callout.
  • Cross-fades back to neutral chrome the instant the worker exits — feels like a "mode" not a layout change.

Dark-mode parity & a11y

  • Light: deep accent gradient over title bar; dark: brighter cyan-blue gradient for sufficient lift off the window frame.
  • Caption buttons stay legible (forced white-on-accent fill in both themes).
  • Taskbar thumbnail preview also shows the tinted chrome — visibility even when window is occluded.

Designer's pick

Recommend Option A — Prominent footer band

Option A directly honors the user's "bottom-left is nice but so hidden" feedback: it keeps the exact anchor they like, swaps the 11 pt secondary-text label for a full-width tinted band the moment IsBusy flips true, and brings the most dangerous piece of context (the target drive) up next to the shield. It accomplishes the loudness the user is asking for without inventing a new affordance, without stealing focus from the phase rail/log, and without any layout reflow — the slim caption strip simply grows in place. Option B (chip) is too easy to mistake for a decorative pill and competes with the Log header; Option C (title-bar tint) is striking but expensive (involves AppWindowTitleBar custom theming and may visually clash on Mica/acrylic backdrops, plus the band would still be needed below to carry the target-drive context). Suggested follow-up: ship A as v1; revisit C as a v2 layered enhancement once we've validated the band reads correctly in real builds.
